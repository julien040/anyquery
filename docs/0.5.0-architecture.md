## 1. Current architecture

Facts established by reading the code. These drive every design decision
below; internalize them first.

### 1.1 Plugin RPC

- Plugins are separate processes managed by `hashicorp/go-plugin` in
  **`net/rpc` mode** (gob encoding), *not* gRPC. See `rpc/rpc.go:26`
  (`go_plugin "github.com/hashicorp/go-plugin"`) and the
  `PluginRPCClient`/`PluginRPCServer` pair wrapping `*rpc.Client`
  (`rpc/rpc.go:39-55`). The handshake happens over the child's stdout;
  the RPC traffic itself flows over a Unix socket that go-plugin opens
  (go-plugin behavior, not visible in this repo).
- `ProtocolVersion = 1`, `rpc/rpc.go:29`. go-plugin refuses to talk to a
  plugin built with a different protocol version — this is our breaking
  lever for 0.5.0.
- Both plugin hooks receive a `*go_plugin.MuxBroker`
  (`rpc/rpc.go:49-55`) and **ignore it today**. The MuxBroker multiplexes
  extra bidirectional streams over the same connection (yamux). It is how
  the plugin will call back into the host for the cache API (§2).
- The wire interface is `InternalExchangeInterface`
  (`rpc/interface.go:24-`): `Initialize`, `Query`, `Insert`, `Update`,
  `Delete`, `Close`. **No method takes a context. Nothing can be
  cancelled.** `Query(connectionID, tableIndex, cursorIndex, constraint)`
  returns a chunk of rows plus a `noMoreRows` bool — cursors are
  incremental, a table scan is many `Query` calls.
- Plugin-side dispatch lives in `rpc/plugin.go`. The maps
  `Plugin.cursors` and `Plugin.tableConnection` (`rpc/plugin.go:128-131`)
  are **not mutex-protected**; two concurrent queries hitting the same
  plugin can produce a `fatal error: concurrent map writes` that kills
  the whole plugin process. Fix this in 0.5.0 regardless of anything else.
- One plugin process serves all profiles/connections
  (`rpc.ConnectionPool`, `rpc/rpc.go:178`, keyed by executable path,
  refcounted).

### 1.2 SQLite virtual-table bridge

- `module/module.go` adapts the RPC interface to SQLite vtabs via the
  fork `github.com/julien040/go-sqlite3-anyquery` (v1.19.4). The fork is
  mattn/go-sqlite3 plus vtab extensions (`PartialUpdate`, LIMIT/OFFSET in
  BestIndex, etc.). Bundled SQLite is recent enough to have
  `sqlite3_is_interrupted()` (5 occurrences in `sqlite3-binding.c` of the
  module in the Go module cache) — relevant for §4.
- The driver fork **already interrupts queries when their context is
  cancelled**: `QueryContext`/`ExecContext` spawn a watcher that calls
  `C.sqlite3_interrupt` on `ctx.Done()` (fork `sqlite3.go:2075` and
  `:2193`). Anyquery never benefits because it passes
  `context.Background()` everywhere (§4.1).
- A `SQLiteModule` is registered per (plugin, profile, table). `Create`
  is lazy — the plugin process starts on first query of one of its
  tables (`module/module.go:108-131`).
- **Known wart:** when a second SQLite connection materializes the same
  module, `Create` returns the *same* `*SQLiteTable` created for the
  first connection (`module/module.go:115-119` `moduleInited`). A vtab
  instance shared between two SQLite connections shares `nextCursor`
  without synchronization. §4 fixes this as a side effect.
- Writes are buffered (`insertBuffer`/`updateBuffer`/`deleteBuffer`,
  flushed on `Open`, on size threshold, on a 5s ticker, and on
  `Disconnect` — `module/module.go:349-368,393-418`).
- File readers (`read_csv`, `read_json`, …) are in-process modules in
  `module/` registered in the ConnectHook
  (`namespace/namespace.go:393-400`). `read_csv` (`module/read_csv.go`)
  is the template for `read_xlsx`: parse args with `parseArgs`
  (`module/helper.go:135`), fetch the file with `openMmapedFile` →
  go-getter download cache (`module/helper.go:104-124`), declare schema,
  cursor iterates.

### 1.3 Caching today (what we are replacing)

- The SDK offers `rpc/helper.Cache` — badger DB + gob, TTL, requires an
  encryption key, stored under
  `$XDG_CACHE_HOME/anyquery/plugins/<paths...>`
  (`rpc/helper/cache.go:199-255`).
- **Plugins don't consistently use it.** e.g. the GitHub plugin
  hand-rolls its own badger setup with `md5(token)` as the encryption key
  (`plugins/github/helper.go:50-70`). Every plugin reimplements
  save/load/TTL.
- **The directory-lock workaround is a bug factory**: if the badger dir
  is already locked (second anyquery process, or a crashed one), the SDK
  silently creates `<dir>-<8 random letters>` and uses that
  (`rpc/helper/cache.go:236-244`). Orphaned cache directories accumulate
  and are never cleaned.
- Central clearing is `clear_plugin_cache(name)` = `os.RemoveAll` on the
  XDG path (`namespace/other_functions.go:64-93`). It races with plugins
  that hold the badger dir open (badger keeps writing into deleted
  files on Unix; on Windows the delete fails outright), and it misses the
  random-suffix orphans.

### 1.4 Query pipelines

Two distinct paths run queries; both matter for §4 and §5.

**CLI path** (`controller/query.go:144-159`): a middleware chain —
`middlewareSlashCommand → middlewareDotCommand → middlewarePRQL →
middlewarePQL → middlewareMySQL → middlewareFileQuery → middlewareQuery`
(`controller/middleware.go`). `middlewareFileQuery` parses **every**
query with the vendored vitess parser and re-serializes it
(`controller/middleware.go:487-562`) to rewrite `read_*(...)` table
functions into `CREATE VIRTUAL TABLE` pre-exec statements. The vendored
parser (`other/sqlparser/`, ~22k-line generated `sql.go`, **`sql.y`
grammar present**, goyacc wired via `other/sqlparser/generate.go:21`) is
a *patched* vitess: upstream `TableName` has no `Args` field; this fork
added it so `read_csv('f.csv', header=true)` parses in FROM position.

**MySQL server path** (`namespace/mysql*.go`): a vitess
`mysql.Listener` (one goroutine per client connection) drives `handler`
(`namespace/mysql_handler.go:40`). Each MySQL connection gets its own
`*sql.Conn` pinned from the shared pool
(`namespace/mysql_handler.go:64-115`). The pool has no
`SetMaxOpenConns` limit (`namespace/namespace.go:498-504`), so
connections don't starve each other at the pool level. With
`MustCatchMySQLSpecific` on, every SELECT is parsed, rewritten and
re-serialized (`namespace/mysql_rewriter.go:307-310`). Results are
**fully buffered** into a `sqltypes.Result` before anything is sent to
the client (`namespace/mysql_handler.go:316-323` +
`convertSQLRowsToSQLResult`).

The vendored vitess parser **does** parse `KILL <id>` / `KILL QUERY <id>`
(`other/sqlparser/ast.go:719` `Kill` struct, `analyzer.go:66`
`StmtKill`) — nobody handles it yet.

## 2. Global cache

### 2.1 Decision

The main process owns all plugin cache storage. A plugin chooses, **per
table**, one of three modes, declared in the `DatabaseSchema` it returns
from `Initialize`:

| Mode | Who writes cache code | What is cached | Transport |
|---|---|---|---|
| `CacheModeNone` (default) | — | nothing | — |
| `CacheModeTransparent` | nobody | full result sets, keyed by constraint hash | host-side interception, zero RPC |
| `CacheModeManaged` | plugin author | whatever the plugin wants (rows, tokens, cursors) | RPC callback into the host over the MuxBroker |

Both modes store into badger databases **opened by the host only**, under
a new root, so clearing/inspection/quotas live in one process:

```
$XDG_CACHE_HOME/anyquery/cachev2/<registry>/<plugin>/<profile-hash>/
```

The legacy `$XDG_CACHE_HOME/anyquery/plugins/**` tree (including the
random-suffix orphans of `rpc/helper/cache.go:236-244`) is dead: 0.5.0
deletes it on first run (one-time sweep, log what was removed).

### 2.2 Schema and manifest changes

Add to `rpc.DatabaseSchema` (`rpc/interface.go`):

```go
type CacheMode int8

const (
	CacheModeNone CacheMode = iota
	CacheModeTransparent
	CacheModeManaged
)

type CachePolicy struct {
	Mode CacheMode
	// TTL for transparent entries. 0 in transparent mode = do not cache
	// (equivalent to CacheModeNone); plugins MUST set it explicitly.
	TTLSeconds int64
	// Entries whose gob-encoded size exceeds this are not stored.
	// 0 = default (64 MiB).
	MaxEntryBytes int64
}

type DatabaseSchema struct {
	// ... existing fields ...
	Cache CachePolicy
}
```

gob tolerates the added field for old hosts, but it doesn't matter: the
`ProtocolVersion` bump (§8) already forces plugins and host to move in
lockstep.

### 2.3 Host-side store

One badger DB per (plugin, profile), lazily opened, all handles owned by
a single `cache.Manager` living next to the `rpc.ConnectionPool`
(created in `Namespace.Init`, `namespace/namespace.go:231`). New package
`cache/` at the repo root.

```go
// package cache
type Manager struct {
	mu     sync.Mutex
	stores map[storeKey]*Store // storeKey{registry, plugin, profileHash}
	root   string              // $XDG_CACHE_HOME/anyquery/cachev2
	masterKey []byte           // see §2.6
}

type Store struct {
	db       *badger.DB
	inMemory bool // true when we fell back (see below)
}
```

Rules that fix today's bugs:

1. **Never create random-suffix directories.** If `badger.Open` fails
   with the directory-lock error (another anyquery process owns it),
   open an **in-memory** badger (`badger.DefaultOptions("").WithInMemory(true)`)
   instead, log one warning. The cache is an optimization; a second
   process silently running uncached is correct. (This replaces
   `rpc/helper/cache.go:236-244`.)
2. **Cross-process clear via tombstone.** `Clear` from a process that
   does not hold the lock writes an empty file `<dir>/.clear-requested`.
   Every `Store` checks the tombstone (a) at open and (b) at most once
   every 30 s on the read path (stat is cheap; store the last-check
   timestamp in the `Store`). When seen: `db.DropAll()`, remove the
   tombstone.
3. **Quota/inspection**: `Manager.Stats()` walks `stores` +
   on-disk dirs and returns per-plugin sizes (badger `db.Size()` for
   open stores, `filepath.WalkDir` for closed ones).

New CLI surface (`cmd/`): `anyquery cache list` (plugin, profile, size,
entry count) and `anyquery cache clear [plugin]`. Rewrite
`clear_plugin_cache` (`namespace/other_functions.go:64`) to call
`Manager.Clear` instead of `os.RemoveAll` (keep the legacy-path removal
too, for the transition).

### 2.4 Transparent mode

Interception point: wrap the RPC client, not the module. Define in
`rpc/` (or `cache/`) a decorator implementing `InternalExchangeInterface`
and wrap it where the module talks to the plugin
(`module/module.go:135` uses `m.client.Plugin.<Method>`; give
`InternalClient` a cached, wrapped `Plugin` field at construction in
`rpc/rpc.go:245-262`).

```go
type cachingClient struct {
	inner rpc.InternalExchangeInterface
	store *cache.Store
	// schema per (connectionID, tableIndex), captured from Initialize,
	// to know each table's CachePolicy
	policies sync.Map // map[tableKey]rpc.CachePolicy

	mu       sync.Mutex
	inflight map[cursorKey]*pendingEntry // rows accumulated per live cursor
	serving  map[cursorKey]*servingState // read offset per cache-hit cursor
}

func (c *cachingClient) Query(connID, tblIdx, curIdx int, q rpc.QueryConstraint) ([][]any, bool, error) {
	pol := c.policy(connID, tblIdx)
	if pol.Mode != rpc.CacheModeTransparent || pol.TTLSeconds <= 0 {
		return c.inner.Query(connID, tblIdx, curIdx, q)
	}
	key := cacheKey(connID, tblIdx, q.Hash()) // see below
	ck := cursorKey{connID, tblIdx, curIdx}

	// 1. Cache-hit path: serve chunks from the stored entry.
	if st := c.servingFor(ck, key); st != nil {
		rows, done := st.nextChunk(4096)
		return rows, done, nil
	}
	// 2. Miss: forward, accumulate, commit when the cursor is exhausted.
	rows, noMore, err := c.inner.Query(connID, tblIdx, curIdx, q)
	if err != nil {
		c.dropInflight(ck)
		return rows, noMore, err
	}
	c.accumulate(ck, key, rows, pol.MaxEntryBytes)
	if noMore {
		c.commit(ck, key, pol) // Set(key, allRows) with TTL; skip if over MaxEntryBytes
	}
	return rows, noMore, nil
}
```

Key: `sha256(pluginPath | connectionID | tableIndex | constraint.Hash())`.
`QueryConstraint.Hash()` (`rpc/interface.go:552-595`) already
normalizes column/order constraints **and includes Limit and Offset** —
so `LIMIT 10` and `LIMIT 100` are distinct entries and there is no
partial-result-poisoning problem. It hashes hidden-parameter constraints
too (they are ordinary column constraints), so
`read`-style parameterized tables key correctly.

Invalidation:

- `Insert`/`Update`/`Delete` on a table → `store.DropPrefix` of that
  table's key prefix (so make the key literal
  `t/<connID>/<tblIdx>/<hash>`; badger `DropPrefix` exists). Do it in the
  same `cachingClient` decorator, *before* forwarding the write.
- TTL is enforced by badger's `WithTTL` on the entry.
- `noMoreRows` never arriving (cursor abandoned mid-scan because SQLite
  stopped early — a `LIMIT` implemented SQLite-side, or an error): the
  inflight accumulation must be dropped, never committed. Hook: cursors
  are reset via a new `Filter` call with the same `cursorIndex`
  (`module/module.go:852-879` `resetCursor` bumps the index, so in
  practice a fresh cursorIndex arrives) — evict `inflight` entries by
  LRU (cap the map at e.g. 64 pending cursors) rather than trying to
  detect abandonment precisely.

Edge cases (write tests for each):

1. Two cursors, same key, concurrently: both accumulate independently;
   last commit wins. Acceptable; do NOT add singleflight in v1.
2. Entry bigger than `MaxEntryBytes`: stop accumulating, mark the
   inflight entry poisoned so it is not committed.
3. Plugin returns rows then errors: dropped by the error path above.
4. Same table, different profiles: different `connectionID` → different
   key. Correct, since config (API token) differs.
5. Clock: TTL uses badger's internal expiry; no wall-clock code needed.

### 2.5 Managed mode (RPC callback over the MuxBroker)

hashicorp/go-plugin's netrpc mode gives both sides the broker. Serving a
host service to the plugin is a **host-allocates-ID, plugin-dials**
handshake:

Host side — `rpc/rpc.go`. Keep the broker at dispense time:

```go
func (p *InternalPlugin) Client(b *go_plugin.MuxBroker, c *rpc.Client) (interface{}, error) {
	return &PluginRPCClient{client: c, broker: b}, nil
}
```

When a connection initializes, allocate a stream ID, serve the cache
service on it, and hand the ID to the plugin inside `InitializeArgs`:

```go
type InitializeArgs struct {
	ConnectionID int
	TableIndex   int
	Config       PluginConfig
	CacheBrokerID uint32 // NEW: 0 = host offers no cache service
}

func (m *PluginRPCClient) Initialize(connID, tblIdx int, config PluginConfig) (DatabaseSchema, error) {
	id := m.broker.NextId()
	// AcceptAndServe registers the value under the RPC name "Plugin",
	// so plugin-side calls are "Plugin.Get", "Plugin.Set", ...
	go m.broker.AcceptAndServe(id, &CacheRPCServer{store: m.storeFor(connID)})
	args := &InitializeArgs{ConnectionID: connID, TableIndex: tblIdx,
		Config: config, CacheBrokerID: id}
	var resp DatabaseSchema
	err := m.client.Call("Plugin.Initialize", args, &resp)
	return resp, err
}
```

`CacheRPCServer` is a plain net/rpc service (single-arg struct in,
pointer out — same style as `PluginRPCServer`, `rpc/rpc.go:139-165`):

```go
type CacheGetArgs struct{ Key string }
type CacheGetReturn struct {
	Rows     [][]interface{}
	Metadata map[string]interface{}
	Found    bool
}
type CacheSetArgs struct {
	Key        string
	Rows       [][]interface{}
	Metadata   map[string]interface{}
	TTLSeconds int64
}

func (s *CacheRPCServer) Get(a *CacheGetArgs, r *CacheGetReturn) error
func (s *CacheRPCServer) Set(a *CacheSetArgs, _ *struct{}) error
func (s *CacheRPCServer) Delete(a *CacheDeleteArgs, _ *struct{}) error
func (s *CacheRPCServer) ClearPrefix(a *CacheClearPrefixArgs, _ *struct{}) error
```

Plugin side — `rpc/plugin.go` must stash the broker
(`InternalPlugin.Server(b *go_plugin.MuxBroker)` receives it,
`rpc/rpc.go:49-51`; today it is discarded — store it in the
`InternalPlugin` and thread it into `internalInterface`). The SDK then
exposes it on `TableCreatorArgs`:

```go
type TableCreatorArgs struct {
	UserConfig   PluginConfig
	TableIndex   int
	ConnectionID int
	Cache        *HostCache // NEW: nil if the host offered no cache
}

// dialed lazily on first use
func (h *HostCache) Get(key string) ([][]interface{}, map[string]interface{}, bool, error)
func (h *HostCache) Set(key string, rows [][]interface{}, meta map[string]interface{}, ttl time.Duration) error
func (h *HostCache) Delete(key string) error
func (h *HostCache) ClearPrefix(prefix string) error

// internal:
conn, err := broker.Dial(args.CacheBrokerID)
client := rpc.NewClient(conn)
client.Call("Plugin.Get", &CacheGetArgs{Key: key}, &ret)
```

Host namespaces managed-mode keys under `m/<connID>/` inside the same
per-plugin store, so `anyquery cache clear <plugin>` wipes both modes
and the transparent invalidation prefix (`t/…`) never collides.

Serialization cost note: managed-mode `Set` ships rows over the local
socket once more than today's in-plugin cache. That is acceptable —
gob over a Unix socket moves hundreds of MB/s, and plugins that cache
per-row API data ship the same bytes over the main channel anyway.
**Do not** try to be clever with shared memory in 0.5.0.

Migration for plugin authors: replace `helper.NewCache(...)` with
`args.Cache`. Keep `rpc/helper.Cache` compiling but make `NewCache`
return an error directing to the new API (the SDK bump is breaking
anyway), and delete its badger dependency in a follow-up.

### 2.6 Encryption

Today each plugin invents a key (GitHub uses `md5(token)` —
`plugins/github/helper.go:52-63`). Centralize:

- Generate a random 32-byte master key on first run; store it in the
  existing config database (`controller/config/`, new single-row table).
- Per-plugin store key = `HKDF-SHA256(master, info=registry+plugin)`
  (`golang.org/x/crypto/hkdf`, already an indirect dep via
  `golang.org/x/crypto`).
- Pass it to `badger.Options.WithEncryptionKey` +
  `WithIndexCacheSize(...)` (badger requires an index cache when
  encryption is on — the old code already knew this,
  `rpc/helper/cache.go:230`).

This is strictly better than the status quo and requires zero plugin
code.

### 2.7 Implementation checklist

1. `cache/` package: `Manager`, `Store`, tombstone protocol, in-memory
   fallback, stats. Unit tests incl. two-manager lock contention (spawn
   the second manager in the same test process — badger's dir lock is
   flock-based and works in-process).
2. `rpc/interface.go`: `CachePolicy` on `DatabaseSchema`;
   `InitializeArgs.CacheBrokerID`; cache RPC arg/return types. Register
   any new concrete types used inside `interface{}` values with
   `gob.Register` in both directions (mirror what `rpc/helper/helper.go`
   does today).
3. `rpc/rpc.go`: keep the broker; serve `CacheRPCServer` per Initialize.
4. `rpc/plugin.go`: stash broker; `TableCreatorArgs.Cache`; `HostCache`.
5. `cachingClient` decorator + wiring in `rpc.ConnectionPool.NewClient`.
6. CLI `anyquery cache list|clear`; rewrite `clear_plugin_cache`;
   legacy-dir sweep.
7. Port two first-party plugins (github, hackernews) to `args.Cache` /
   transparent mode as the reference implementations.

## 3. `read_xlsx` module

An in-repo SQLite module like `read_csv`, **plus write support** —
which none of the existing `read_*` modules have, so this is also the
reference implementation for writable file modules.

Library: `github.com/xuri/excelize/v2` (verified at **v2.11.0**; not yet
in `go.mod`). `.xls` (OLE) is **not** supported by excelize; `.xlsx`,
`.xlsm`, `.xltx`, `.xltm`, `.xlam` are.

### 3.1 User-facing surface

```sql
SELECT * FROM read_xlsx('report.xlsx');
SELECT * FROM read_xlsx('report.xlsx', sheet='Q3', header=true);
SELECT * FROM read_xlsx('report.xlsx', range='B2:F100');
INSERT INTO x VALUES (...);      -- via CREATE VIRTUAL TABLE x USING xlsx_reader(...)
UPDATE x SET price = 2 WHERE row_number = 5;
DELETE FROM x WHERE row_number = 5;
```

Arguments (parse with `parseArgs`, `module/helper.go:135`, same aliases
style as `module/read_csv.go:92-111`):

| arg | default | meaning |
|---|---|---|
| `file`/`path`/`url`/`src` (or positional arg 3) | required | local path or remote URL |
| `sheet` | first worksheet in `GetSheetList()` order that is visible | sheet name (excelize matches case-insensitively) |
| `range` | whole used range | A1-style rectangle, e.g. `B2:F100` |
| `header` | `false` | first data row provides column names |
| `password` | empty | for encrypted workbooks (`excelize.Options{Password: ...}`) |
| `fill_merged` | `false` | repeat merged-cell anchor values across the range |
| `schema` | empty | explicit `CREATE TABLE` override, exactly like read_csv (`module/read_csv.go:152-188`) |

Registration:

- `conn.CreateModule("xlsx_reader", &module.XlsxModule{Restrictions: n.restrictions})`
  next to the others in `namespace/namespace.go:393-400`.
- `"read_xlsx": "xlsx_reader"` in `supportedTableFunctions`
  (`controller/middleware.go:454-464`) — and in the new parser path (§5).

### 3.2 Reading design

Fetch: reuse the download cache but **not** mmap — excelize wants a path
or reader, and for writes we need a real local file. Split
`openMmapedFile` (`module/helper.go:104`): extract a
`downloadToCache(src, r *Restrictions) (localPath string, err error)`
that returns the cached file path; read_xlsx calls
`excelize.OpenFile(localPath, opts...)`.

**Writes are only allowed when the source is a local path** (no URL
schemes, no stdin). Detect: after `downloadToCache`, writable iff the
resolved path == the original argument cleaned (i.e. it wasn't fetched
into the download cache). Declare the table read-only otherwise (the
vtab's write methods return an explicit error).

Schema inference (Connect):

1. Open workbook; resolve sheet; parse `range` if given
   (`excelize.CellNameToCoordinates` twice).
2. Stream the first row (or the first row of `range`) for column names
   when `header=true`, else `col0..colN`. Sanitize with
   `transformSQLiteValidName` (`module/helper.go:163`) and dedupe by
   suffixing `_2`, `_3`, … (read_csv doesn't dedupe — do it here; two
   identical header cells otherwise produce a DeclareVTab error, cf. the
   duplicate check in `module/module.go:224`).
3. Type inference: scan up to 25 data rows with
   `rows.Columns(excelize.Options{RawCellValue: true})`. Per column:
   all-parse-as-int → INTEGER; all float → REAL; all `0/1/TRUE/FALSE` →
   BOOLEAN; else TEXT. **Date detection cannot use the stream** (the
   iterator exposes no per-cell style), so for each column whose raw
   values are numeric, check the style of ONE sample cell:
   `styleID, _ := f.GetCellStyle(sheet, cell); style, _ := f.GetStyle(styleID)`
   and treat as DATETIME when `style.NumFmt` is one of the OOXML
   built-in date ids (14–22, 27–36, 45–47, 50–58) or
   `style.CustomNumFmt` contains date tokens (`[dmhysDMHYS]` outside
   quoted sections). This id list is spec knowledge, not an excelize
   API — keep it in one function with a comment.
4. Declare schema; **always include a hidden-less first column**
   `row_number INTEGER` — it is the primary key for writes (§3.3) and
   maps to the spreadsheet row. Do NOT mark it HIDDEN; users need it.

Cursor (one `*excelize.Rows` per cursor — documented concurrency-safe):

```go
func (t *XlsxTable) Open() (sqlite3.VTabCursor, error) {
	rows, err := t.file.Rows(t.sheet)
	...
}
```

- `rows.Next()` yields **every** row index including interior empty
  rows; for a fully empty row `Columns()` returns `(nil, nil)`. Skip
  rows before the range start / header row; stop after range end.
- `Columns()` slices are **ragged**: trailing empty cells are dropped.
  Column accessor must NULL-pad (read_csv already does this pattern,
  `module/read_csv.go:308-311`).
- Values arrive as strings. Conversion per declared column type:
  INTEGER/REAL → `strconv`; BOOLEAN → accept `1/0/TRUE/FALSE`;
  DATETIME → the raw value is the Excel serial number; convert with
  `excelize.ExcelDateToTime(serial, date1904)` where
  `date1904 = props.Date1904 != nil && *props.Date1904` from
  `f.GetWorkbookProps()`, then emit RFC3339. On any parse failure emit
  NULL (read_csv convention).
- Formula cells: the raw value is the **cached** result stored by the
  last writer. If a producer saved without cached values you get `""` →
  NULL. Do not call `CalcCellValue` in the scan path (best-effort,
  slow); acceptable v1 limitation, document it.
- `fill_merged=true`: load `f.GetMergeCells(sheet)` once at Connect into
  a lookup (cell→anchor value) and overlay in the Column accessor. The
  streaming reader does NOT resolve merges by itself (non-anchor cells
  read `""`).
- Rowid = spreadsheet row number. Deterministic, survives re-scans,
  needed for UPDATE/DELETE.

### 3.3 Write design

Model: mutate the in-memory `*excelize.File`, save debounced +
at safe points, atomically.

- The vtab methods to implement are the same trio the plugin bridge
  implements (`module/module.go:450-521`): `Insert(id any, vals []any)
  (int64, error)`, `Update(id any, vals []any) error`,
  `Delete(id any) error`, plus `PartialUpdate() bool` → `false`.
- **INSERT**: append at `lastRow+1` via
  `f.SetSheetRow(sheet, cellOfRow(next), &vals)` (pointer-to-slice is
  required by excelize). Track `lastRow` in the table struct; return the
  new row number as rowid. A NULL `row_number` in the insert is the
  normal case; a user-specified `row_number` targeting an existing row
  is an error (`UNIQUE constraint`-style message).
- **UPDATE**: `id` is the row number. Write each non-nil column with
  `f.SetCellValue`. Type mapping is natural (excelize accepts Go ints,
  floats, string, bool, `time.Time`).
- **DELETE**: **do not call `RemoveRow` immediately.** `RemoveRow`
  shifts all subsequent rows up, which corrupts row-number rowids that
  SQLite has already collected for the same statement (e.g.
  `DELETE FROM x WHERE price > 3` deletes several rows). Buffer deleted
  row numbers in the table struct; apply them **sorted descending** at
  flush time. This mirrors the plugin-bridge delete buffer
  (`module/module.go:502-521`).
- **Flush points** (same trio as the plugin bridge, `module/module.go:349`):
  at the start of every `Open()` (so a SELECT after a write sees the
  writes and the row numbering is settled), on `Disconnect`, and on a
  ticker after ~5 s of write inactivity. Flush = apply buffered deletes
  descending → save atomically:

```go
func (t *XlsxTable) saveAtomic() error {
	t.writeMu.Lock()
	defer t.writeMu.Unlock()
	dir, base := filepath.Split(t.path)
	tmp, err := os.CreateTemp(dir, "."+base+".tmp-*")
	if err != nil { return err }
	defer os.Remove(tmp.Name())
	if err := t.file.Write(tmp); err != nil { tmp.Close(); return err }
	if err := tmp.Sync(); err != nil { tmp.Close(); return err }
	if err := tmp.Close(); err != nil { return err }
	if fi, err := os.Stat(t.path); err == nil {
		os.Chmod(tmp.Name(), fi.Mode()) // CreateTemp gives 0600
	}
	return os.Rename(tmp.Name(), t.path) // atomic on same filesystem
}
```

  `SaveAs` truncates in place (`O_TRUNC` then write) — a crash corrupts
  the workbook. Never use it.
- **Concurrency**: guard all mutations and `saveAtomic` with one
  `sync.Mutex` in the table. Reads via `Rows` are documented
  concurrency-safe against `SetCellValue`/`SetSheetRow`, but do not rely
  on reading your own uncommitted deletes — the Open()-flush rule
  guarantees scans only start from a settled file.

### 3.4 Edge cases to test

1. Ragged rows, interior empty rows, leading empty rows/cols (data NOT
   at A1 — positions are preserved by excelize; the range logic must not
   assume A1).
2. Real `.xls` input: excelize sees the OLE magic and fails with
   `ErrWorkbookFileFormat` (it thinks it's an encrypted xlsx). Catch
   this error and return
   `"this looks like a legacy .xls file; only .xlsx is supported"`.
3. 1900 vs 1904 date systems; a date column mixed with a text cell.
4. Formula whose cached value is empty.
5. Merged cells with and without `fill_merged`.
6. Sheet names with spaces/unicode; wrong sheet name (list available
   sheets in the error); chartsheets (appear in `GetSheetList`; opening
   `Rows` on one errors — surface it cleanly).
7. `header=true` with duplicate header cells.
8. INSERT then SELECT in one session (flush-on-Open); DELETE of several
   rows in one statement (descending apply); UPDATE of a row already
   deleted in the same buffer (define: delete wins).
9. Password-protected workbook, wrong password.
10. Very wide sheet (one cell in `XFD1` makes 16k-wide rows — cap
    declared columns at the header/range width).
11. Save with the file deleted underneath / directory unwritable.
12. Memory: every save serializes the whole zip in RAM
    (`WriteToBuffer`) — document that write support targets
    human-scale workbooks, not GB-scale exports. Reads stream (O(row)
    memory; oversized XML parts spill to temp files automatically).

## 4. Concurrency: cancellable queries

### 4.1 Diagnosis (all verified)

The complaint: “anyquery blocks on a query and nothing else runs in
MySQL-server mode.” Five concrete causes, in decreasing severity:

1. **Nothing is cancellable.** Every DB call uses a background context:
   `namespace/mysql_handler.go:317` and `:325`
   (`conn.QueryContext(context.Background(), ...)`), CLI
   `controller/middleware.go:342` (`DB.Query`, no ctx). The driver fork
   would interrupt SQLite on ctx cancel (fork `sqlite3.go:2075,2193`) —
   we just never hand it a cancellable context. `KILL` is parseable
   (§1.4) but unhandled, so a stuck client cannot even be killed from a
   second connection.
2. **A blocking plugin RPC cannot be interrupted at all.**
   `sqlite3_interrupt` only takes effect *between* vtab callbacks; the
   scan is usually *inside* `requestRowsFromPlugin`
   (`module/module.go:886-924`), a synchronous gob RPC that returns
   whenever the plugin's HTTP call returns. Worst case (plugin hangs
   forever) the SQLite statement never regains control.
3. **Ctrl-C in the CLI makes it worse**: the signal handler does
   `db.Close(); os.Exit(0)` (`controller/query.go:135-141`).
   `db.Close()` waits for the in-flight connection → the process
   *appears* hung right when the user tries to escape. (The shell also
   binds Ctrl-C to “insert `.quit`” — `controller/shell.go:361-367` —
   which only applies between queries.)
4. **The MySQL handler buffers entire result sets**
   (`namespace/mysql_handler.go:316-323`) before sending row one. A
   10M-row scan looks frozen for its whole duration and holds all rows
   in RAM. Most GUI clients use ONE connection for everything, so a slow
   query freezes the whole client even though the server could accept
   other connections.
5. **Plugin-side data races** under concurrent queries (unprotected maps,
   `rpc/plugin.go:128-131`) can crash the plugin process, which then
   breaks every table of that plugin until restart.

Not guilty: the connection pool (no `SetMaxOpenConns(1)` anywhere;
WAL mode is on, `namespace/namespace.go:199`; busy_timeout defaults to
5000 ms in the driver).

Scope decision for 0.5.0: **cancellation-first**. Parallelism across
connections mostly already exists; we make every query killable and
every wait bounded. Streaming results (fix #4) is a stretch goal.

### 4.2 Per-query contexts + KILL (MySQL server mode)

In `handler` (`namespace/mysql_handler.go:40`):

```go
type handler struct {
	// ... existing ...
	runningQueries   map[uint32]context.CancelFunc // keyed by MySQL connection ID
	runningQueriesMu sync.Mutex
	QueryTimeout     time.Duration // 0 = none; new --query-timeout flag
}

func (h *handler) beginQuery(connID uint32) (context.Context, func()) {
	ctx := context.Background()
	var cancel context.CancelFunc
	if h.QueryTimeout > 0 {
		ctx, cancel = context.WithTimeout(ctx, h.QueryTimeout)
	} else {
		ctx, cancel = context.WithCancel(ctx)
	}
	h.runningQueriesMu.Lock()
	h.runningQueries[connID] = cancel
	h.runningQueriesMu.Unlock()
	return ctx, func() {
		h.runningQueriesMu.Lock()
		delete(h.runningQueries, connID)
		h.runningQueriesMu.Unlock()
		cancel()
	}
}
```

`runQuery` (`namespace/mysql_handler.go:250`) calls `beginQuery` and
threads `ctx` into `runSimpleQuery` → `conn.QueryContext(ctx, ...)` /
`ExecContext(ctx, ...)`. The driver fork then interrupts SQLite on
cancel; the statement aborts with `interrupted` at the next vtab
callback boundary.

KILL handling in `runQueryWithMySQLSpecific`
(`namespace/mysql_rewriter.go:288`):


treat both as query-cancel in v1.) Also handle the
`RewriteMySQLQueries == false` path: `runSimpleQuery` should
prefix-match `KILL ` before shipping it to SQLite.

For discoverability implement `SHOW PROCESSLIST` in

Connection teardown: `ConnectionClosed`

### 4.4 Interrupting the plugin RPC (protocol v2)

Two cooperating halves:

**Host half — stop waiting.** `requestRowsFromPlugin`
(`module/module.go:886`) becomes an async call that also watches the
interrupt flag of its SQLite connection:

```go
// fork addition (julien040/go-sqlite3-anyquery):
// func (c *SQLiteConn) IsInterrupted() bool {
//     return C.sqlite3_is_interrupted(c.db) != 0   // SQLite >= 3.41, present in the bundled amalgamation
// }

call := cursor.client.Plugin.QueryAsync(...)   // wraps net/rpc client.Go
tick := time.NewTicker(100 * time.Millisecond)
defer tick.Stop()
for {
	select {
	case <-call.Done:
		return call.Rows, call.NoMoreRows, call.Err
	case <-tick.C:
		if cursor.conn.IsInterrupted() {
			// tell the plugin to stop, then abandon the in-flight call
			go cursor.client.Plugin.Interrupt(cursor.connectionIndex, cursor.tableIndex, cursor.cursorIndex)
			return nil, false, errors.New("query interrupted")
		}
	}
}
```

To have `cursor.conn`, the cursor needs the `*sqlite3.SQLiteConn` it
belongs to. `Create`/`Connect` receive it (`module/module.go:108`);
store it on the table and hand it to cursors in `Open`. **This forces
fixing the shared-vtab wart** (§1.2): `Create` must return a fresh
`*SQLiteTable` per SQLite connection (sharing the `InternalClient` and
plugin `Initialize` result via the module, which stays the cache for
schema + client). The abandoned net/rpc call is harmless: the reply is
drained by the rpc client when it eventually arrives, and the cursor
index is never reused (each Filter/reset takes a fresh index,
`module/module.go:1062-1067`).

**Plugin half — actually stop working.** New RPC method + optional SDK
interface:

```go
// rpc/interface.go
type InternalExchangeInterface interface {
	// ... existing ...
	Interrupt(connectionID int, tableIndex int, cursorIndex int) error
}

// rpc/plugin.go — plugin SDK, opt-in for plugin authors:
type ReaderInterfaceContext interface {
	QueryContext(ctx context.Context, constraint QueryConstraint) ([][]interface{}, bool, error)
}
```

`internalInterface` keeps `map[cursorKey]context.CancelFunc`; `Query`
creates a per-cursor context; `Interrupt` cancels it. Readers
implementing `ReaderInterfaceContext` get the context (pass it to HTTP
calls); plain `ReaderInterface` readers keep working — cancellation then
only takes effect between chunks, which is still a strict improvement.
While in there, **add the missing mutex** around `Plugin.cursors` /
`tableConnection` (diagnosis #5).

Bump `ProtocolVersion` to 2 (`rpc/rpc.go:29`) — same bump that carries
the cache changes (§8).

### 4.5 Stretch: stream results to MySQL clients

`ComQuery`'s callback can be invoked multiple times (vitess streams

## 5. SQL parser

### 5.1 The real problem is silent corruption, not parse failures

Queries that vitess *cannot* parse already fall through untouched
(`controller/middleware.go:495-503` returns early on parse error;
`namespace/mysql_rewriter.go:317-319` runs `StmtUnknown` raw). The
damage comes from queries that vitess parses **as MySQL** and then
re-serializes with different semantics:

- `controller/middleware.go:562` — `middlewareFileQuery` re-serializes
  **every parseable query**, even when no `read_*` function was found.
- `namespace/mysql_rewriter.go:307-310` — server mode re-serializes
  **every SELECT**.

Concrete corruptions (turn each into a regression test):

| Input (valid SQLite) | What vitess does | Result |
|---|---|---|
| `SELECT a \|\| b FROM t` | parses `\|\|` as logical OR (MySQL default) | **silently wrong data** — concat becomes boolean |
| `SELECT "name" FROM t` | `"..."` is a string literal in MySQL | selects the constant `'name'`, not the column |
| `... WHERE x GLOB 'a*'` | parse error | ok (raw passthrough) — but only by luck |
| `INSERT ... ON CONFLICT DO UPDATE` | parse error | ok (passthrough) |
| `SELECT x -> '$.a' FROM t` | parses (MySQL has `->`) | usually round-trips; verify |

### 5.2 Plan overview (agreed with maintainer)

1. **Immediate correctness fix, no new dependency**: never re-serialize
   unless the rewrite actually changed the AST.
2. **Adopt `github.com/rqlite/sql`** — a real, maintained, pure-Go
   SQLite parser — for the SQLite-facing paths (CLI pipeline). Verified
   viable; details below.
3. Server mode keeps vitess (MySQL clients genuinely speak MySQL), with
   fix (1) applied to `rewriteSelectStatement`.
4. Grammar-patching the vendored vitess (`other/sqlparser/sql.y` exists,
   goyacc regeneration is wired in `other/sqlparser/generate.go:21`)
   remains the fallback if (2) hits a wall — it is feasible but each
   patch is yacc surgery on a 22k-line generated file; avoid unless
   forced.

### 5.3 Fix 1 — reserialize only on change

`middlewareFileQuery`: the rewrite closure already knows when it
replaced a node (`cursor.Replace(...)`,
`controller/middleware.go:548`). Hoist a `modified bool`, set it there,
and guard the final assignment:

```go
modified := false
rewrite := func(cursor *sqlparser.Cursor) bool {
	...
	cursor.Replace(sqlparser.NewTableName(tableName))
	modified = true
	return true
}
sqlparser.Rewrite(stmt, nil, rewrite)
...
if modified {
	queryData.SQLQuery = sqlparser.String(stmt)
}
```

Same pattern in `rewriteSelectStatement`
(`namespace/mysql_rewriter.go:557`): thread a `*bool` through its
rewrites (variable/function remapping) and only call
`sqlparser.String` in `runQueryWithMySQLSpecific` when something
changed; otherwise run the original text. This alone fixes the `||` and
`"identifier"` corruption for every pure-SQLite query on both paths.

### 5.4 Fix 2 — rqlite/sql on the CLI path

`github.com/rqlite/sql` (fork of benbjohnson/sql maintained by the
rqlite project; **no semver tags — pin a pseudo-version and treat
upgrades deliberately**; MIT). Verified against commit `1b2524a`
(2026-02-23) by compiling probes:

- API: `sql.NewParser(strings.NewReader(q)).ParseStatement()`
  (`(nil, io.EOF)` on empty input), `ParseStatements()` for
  multi-statement input. AST nodes all implement `String()`;
  `sql.Walk(visitor, node)` is a **rewriting** walk (`VisitFunc`
  adapter returns a replacement node).
- **Table-valued functions in FROM — the feature this whole path needs —
  work**, including named arguments:
  `SELECT * FROM read_csv('f.csv', header=true)` parses to
  `*sql.QualifiedTableFunctionName{Name, Args, Alias}`; `header=true`
  arrives as a `BinaryExpr(EQ, Ident, BoolLit)`.
- Coverage verified: CTEs (incl. MATERIALIZED), window functions,
  UPSERT, `->`/`->>`, PRAGMA, EXPLAIN, generated columns, compound
  selects, bind params, CREATE VIRTUAL TABLE, triggers, UPDATE…FROM.
- Verified gaps and hazards (each needs a guard):
  1. `String()` on `UpdateStatement`/`DeleteStatement` **drops
     RETURNING**; only `InsertStatement` keeps it.
  2. `String()` on `CreateTableStatement` **drops STRICT and
     WITHOUT ROWID**.
  3. `LIKE x ESCAPE y` parses but **`String()` panics**.
  4. No zero-argument TVFs (`my_func()`) — irrelevant, `read_*` always
     has arguments. No schema-qualified TVFs — irrelevant.
  5. `ATTACH`/`DETACH`/`VACUUM` are not parseable statements — they must
     fall through raw.
  6. `String()` normalizes aggressively (all identifiers double-quoted,
     keywords uppercased) — fine for SQLite, but it means the rewritten
     text never equals the input; only emit it when we actually changed
     the AST (same rule as fix 1).

New package `other/sqliteparser` wrapping it:

```go
package sqliteparser

// Result of the read_* rewrite.
type Rewritten struct {
	Query    string   // rewritten SQL — only valid when Changed
	PreExec  []string // CREATE VIRTUAL TABLE ...
	PostExec []string // DROP TABLE ...
	Changed  bool
}

// RewriteTableFunctions replaces read_*(...) table functions with
// temporary virtual tables. Returns Changed=false (and no error) when
// the query has no read_* function or cannot be safely rewritten.
func RewriteTableFunctions(query string, supported map[string]string) (Rewritten, error) {
	stmt, err := sql.NewParser(strings.NewReader(query)).ParseStatement()
	if err != nil {
		return Rewritten{}, err // caller falls back
	}
	if !safeToReserialize(stmt) { // guards for hazards 1-3 below
		return Rewritten{}, nil
	}
	var out Rewritten
	node, err := sql.Walk(sql.VisitFunc(func(n sql.Node) (sql.Node, error) {
		fn, ok := n.(*sql.QualifiedTableFunctionName)
		if !ok {
			return n, nil
		}
		module, ok := supported[strings.ToLower(fn.Name.Name)]
		if !ok {
			return n, nil
		}
		tmp := randomTableName()
		var args []string
		for _, a := range fn.Args {
			args = append(args, a.String()) // 'f.csv', "header" = TRUE
		}
		out.PreExec = append(out.PreExec, fmt.Sprintf(
			"CREATE VIRTUAL TABLE %s USING %s(%s);", tmp, module, strings.Join(args, ", ")))
		out.PostExec = append(out.PostExec, "DROP TABLE "+tmp+";")
		out.Changed = true
		repl := &sql.QualifiedTableName{Name: &sql.Ident{Name: tmp}}
		if fn.Alias != nil {
			repl.Alias = fn.Alias
		}
		return repl, nil
	}), stmt)
	if err != nil || !out.Changed {
		return Rewritten{}, err
	}
	out.Query = safeString(node) // recover() around String(), hazard 3
	if out.Query == "" {
		return Rewritten{}, nil // String() panicked → give up, run raw
	}
	return out, nil
}
```

`safeToReserialize`: return false for `*sql.UpdateStatement` /
`*sql.DeleteStatement` with a RETURNING clause and for
`*sql.CreateTableStatement` with STRICT/WITHOUT ROWID (hazards 1–2) —
those queries run raw (a `read_*` inside them is vanishingly rare; the
error message from SQLite will say the table doesn't exist, acceptable).
`safeString` wraps `node.String()` in `defer recover()` (hazard 3).

Note the arg serialization: rqlite/sql prints named args as
`"header" = TRUE`. `parseArgs` in `module/helper.go:133` matches
`key=value` with optional quotes around the key — `"header" = TRUE`
matches its regex (`^\s*['"`]?([^= '"`]+?)['"`]?\s*=\s*...`). Add a test
to lock this in.

`middlewareFileQuery` becomes:

```go
rw, err := sqliteparser.RewriteTableFunctions(queryData.SQLQuery, supportedTableFunctions)
switch {
case err == nil && rw.Changed:
	queryData.SQLQuery = rw.Query
	queryData.PreExec = append(queryData.PreExec, rw.PreExec...)
	queryData.PostExec = append(queryData.PostExec, rw.PostExec...)
	return true
case err == nil: // parsed fine, nothing to do — run the original text
	return true
default: // rqlite/sql could not parse: fall back to the vitess path
	// (existing code, now with the modified-flag guard from §5.3,
	// covers MySQL-flavored input pasted into the CLI)
}
```

Also route the statement-type probe on the CLI
(`controller/middleware.go:327`, deciding Query vs Exec) through
rqlite/sql first: a plain `switch stmt.(type)` on
`*sql.SelectStatement`, `*sql.ExplainStatement`, … with vitess
`GetQueryType` as fallback. Keep `middlewareMySQL` (SHOW/DESCRIBE
emulation) on vitess — it exists precisely for MySQL syntax.

## 6. Shell input

### 6.1 Decision

Stay on `github.com/elk-language/go-prompt` **v1.3.1** (already the
dependency). It ships everything needed — a `Lexer` interface for
syntax highlighting and a `Completer` — and the current shell simply
doesn't use them (`controller/shell.go:358-404` passes neither). No
fork, no library switch. All signatures below verified against the
v1.3.1 source.

Facts that shape the implementation:

- The completer signature returns the replacement range:
  `type Completer func(prompt.Document) (suggestions []prompt.Suggest, startChar, endChar istrings.RuneNumber)`
  where `istrings` is `github.com/elk-language/go-prompt/strings`.
  Selecting a suggestion **immediately replaces** `[startChar,endChar)`.
- **The completer and the lexer run on essentially every keystroke.**
  The dropdown shows whenever ≥1 suggestion is returned (there is no
  “press Tab to open”; Tab cycles the selection, and Tab with zero
  suggestions inserts indentation). Consequence: the completer must be
  pure in-memory — never query the database inside it.
- Lexer: `type Lexer interface { Init(string); Next() (Token, bool) }`,
  with `prompt.NewEagerLexer(fn LexerFunc)` wrapping a
  `func(string) []prompt.Token`. Tokens carry **inclusive byte**
  offsets: `prompt.NewSimpleToken(firstByte, lastByte, prompt.SimpleTokenWithColor(c))`.
  Uncovered bytes render with the default color (no tokens needed for
  whitespace). Colors are the 16 ANSI colors only (`prompt.Color` enum)
  — no truecolor.
- Enable with `prompt.WithLexer(...)` and `prompt.WithCompleter(...)` in
  the `prompt.New` call at `controller/shell.go:358`.
- The existing multi-line handling (`WithExecuteOnEnterCallback`,
  `controller/shell.go:378-400`) is unaffected and stays.

### 6.2 Syntax highlighter

New file `controller/shell_lexer.go`. Hand-rolled tokenizer — no parser
involved, it must be cheap (full re-lex per keystroke) and must never
fail on partial input:

```go
var sqlKeywords = map[string]struct{}{ /* SELECT FROM WHERE JOIN ... ~60 words
	(SQLite keyword list), lowercase keys */ }

func sqlLexerFunc(line string) []prompt.Token {
	var toks []prompt.Token
	b := []byte(line)
	i := 0
	emit := func(first, last int, color prompt.Color) {
		toks = append(toks, prompt.NewSimpleToken(
			istrings.ByteNumber(first), istrings.ByteNumber(last),
			prompt.SimpleTokenWithColor(color)))
	}
	for i < len(b) {
		switch c := b[i]; {
		case c == '\'' || c == '"' || c == '`':
			j := scanQuoted(b, i, c) // handles doubled-quote escaping; unterminated -> len(b)-1
			emit(i, j, prompt.Green)
			i = j + 1
		case c == '-' && i+1 < len(b) && b[i+1] == '-':
			j := scanUntil(b, i, '\n')
			emit(i, j, prompt.DarkGray)
			i = j + 1
		case c == '/' && i+1 < len(b) && b[i+1] == '*':
			j := scanBlockComment(b, i) // unterminated -> len(b)-1
			emit(i, j, prompt.DarkGray)
			i = j + 1
		case c >= '0' && c <= '9':
			j := scanNumber(b, i)
			emit(i, j, prompt.Yellow)
			i = j + 1
		case isIdentStart(c):
			j := scanIdent(b, i)
			word := strings.ToLower(string(b[i : j+1]))
			if _, ok := sqlKeywords[word]; ok {
				emit(i, j, prompt.Turquoise)
			} // plain identifiers: no token -> default color
			i = j + 1
		case c == '.' && i == indexOfFirstNonSpace(b):
			emit(i, len(b)-1, prompt.Fuchsia) // dot-command line
			i = len(b)
		default:
			i++
		}
	}
	return toks
}
// wire-up: prompt.WithLexer(prompt.NewEagerLexer(sqlLexerFunc))
```

Rules the implementation must respect: byte indexes are **inclusive**
both ends; multi-byte runes must not be split (scan by
`utf8.DecodeRune` inside `scanIdent`/`scanQuoted`); unterminated
strings/comments highlight to end of buffer (this is the common state
while typing — it must not glitch).

### 6.3 Completer

New file `controller/shell_completer.go`. Two pieces:

**A schema snapshot, refreshed outside the completer.** In
`shell.InputQuery` (`controller/shell.go:357`) refresh before showing
the prompt (i.e. once per query cycle, plus once at startup):

```go
type schemaCache struct {
	tables   []prompt.Suggest            // table + vtab module names
	columns  map[string][]prompt.Suggest // lazily filled per table
	dotCmds  []prompt.Suggest            // .tables .mode .output ... (static)
	keywords []prompt.Suggest            // static
	functions []prompt.Suggest           // static: SQLite builtins + anyquery functions
}

func (s *schemaCache) refresh(db *sql.DB) {
	// same queries the dot-commands use (controller/middleware.go:189):
	// SELECT name FROM pragma_table_list UNION SELECT name FROM pragma_module_list
	//   WHERE name NOT LIKE 'fts%' AND name NOT LIKE 'rtree%'
	// columns for tables referenced in the previous query:
	//   SELECT name, type FROM pragma_table_info(?)
}
```

Never call `pragma_table_info` on a **plugin** table eagerly — it
forces the plugin process to start (vtab `Create` fires on first
access, §1.2). Only fetch columns for tables the user has already
referenced in the current buffer, and cache negative results.

**The completer itself** — pure function over the snapshot:

```go
func (s *schemaCache) complete(d prompt.Document) ([]prompt.Suggest, istrings.RuneNumber, istrings.RuneNumber) {
	end := d.CurrentRuneIndex()
	word := d.GetWordBeforeCursor()
	start := end - istrings.RuneCount([]byte(word))

	line := d.TextBeforeCursor()
	if strings.HasPrefix(strings.TrimSpace(d.Text), ".") {
		return prompt.FilterHasPrefix(s.dotCmds, word, true), start, end
	}
	if word == "" {
		return nil, 0, 0 // no popup spam on space/newline
	}
	prev := lastKeywordBefore(line) // scan back for FROM/JOIN/INTO/UPDATE/TABLE/SELECT/WHERE/BY/ON/SET
	var pool []prompt.Suggest
	switch prev {
	case "from", "join", "into", "update", "table":
		pool = s.tables
	case "select", "where", "by", "on", "and", "or", "set", "having":
		pool = append(s.columnsForTablesIn(d.Text), s.functions...)
	default:
		pool = append(append([]prompt.Suggest{}, s.keywords...), s.tables...)
	}
	return prompt.FilterFuzzy(pool, word, true), start, end
}
```

`columnsForTablesIn` regex-scans the buffer for known table names and
concatenates their cached column lists — deliberately dumb; do not
parse SQL here.

### 6.4 Keybind change (UX decision, agreed)

Today Ctrl-C inserts `.quit` (`controller/shell.go:361-367`) — i.e.
Ctrl-C at the prompt exits the shell. Change to the standard:
**Ctrl-C clears the current input line, Ctrl-D (on empty line) exits.**
Keep `.exit`/`.quit`. This composes with §4.3 (Ctrl-C during a running
query cancels the query — that path is outside go-prompt).

## 7. End-to-end testing

Two harnesses, built first because everything else in this release
lands on top of them.

### 7.1 CLI e2e: testscript

`github.com/rogpeppe/go-internal/testscript` (verified **v1.15.0**) —
the txtar-script framework the Go toolchain uses for `cmd/go` tests.
Why it fits: adding a test = adding a small text file; no Go code per
test; each script runs in a fresh temp dir with the fixture files it
declares; assertions are regex on stdout/stderr or golden-file compares
with built-in regeneration.

**Wiring.** `testscript.Main` clones the *test binary itself* under the
command name and dispatches to your `main` — no separate `go build`,
and `main()` gets coverage. Since `main()` lives in the root package
(`main.go:34`), the harness goes in `package main`:

```go
// e2e_test.go — repo root, package main
package main

import (
	"flag"
	"testing"

	"github.com/rogpeppe/go-internal/testscript"
)

var update = flag.Bool("update", false, "update golden files in testscripts")

func TestMain(m *testing.M) {
	testscript.Main(m, map[string]func(){
		"anyquery": main,
	})
}

func TestCLIScripts(t *testing.T) {
	testscript.Run(t, testscript.Params{
		Dir:           "test/e2e/scripts",
		UpdateScripts: *update,
		Setup: func(env *testscript.Env) error {
			// Isolate from the developer's real config/cache: anyquery
			// resolves both via adrg/xdg, which honors these variables.
			env.Setenv("XDG_CONFIG_HOME", env.WorkDir+"/xdg-config")
			env.Setenv("XDG_CACHE_HOME", env.WorkDir+"/xdg-cache")
			env.Setenv("NO_COLOR", "1")
			env.Setenv("FIXTURES", fixturesDir()) // absolute path to test/e2e/fixtures (binary files: xlsx, parquet)
			return nil
		},
		Cmds: map[string]func(*testscript.TestScript, bool, []string){
			"mysql":    cmdMySQL,    // §7.2
			"waitport": cmdWaitPort, // poll until a TCP port accepts, with deadline
		},
	})
}
```

Two gotchas verified against the source: scripts run **in parallel
unconditionally** (each is `t.Parallel()`) — no shared mutable state,
pick per-script ports; and `$HOME` is set to `/no-home`, which is why
the XDG overrides above are load-bearing (adrg/xdg falls back to $HOME
paths otherwise).

**Example script** `test/e2e/scripts/read_csv.txtar`:

```txtar
# read_csv with header and type coercion
exec anyquery -q 'SELECT name, age + 1 FROM read_csv(''people.csv'', header=true) ORDER BY name;' --format csv
cmp stdout expected.csv
! stderr .

# regression §5.1: || must stay string concatenation
exec anyquery -q 'SELECT ''a'' || ''b'';' --plain
stdout '^ab$'

-- people.csv --
name,age
alice,30
bob,25
-- expected.csv --
name,"age + 1"
alice,31
bob,26
```

Run `go test -run TestCLIScripts -update` to regenerate the `cmp`
golden sections in place. Script language cheat sheet (all built-in):
`exec cmd args` / `! exec` (must fail), `stdout 'regex'` /
`stderr 'regex'` (RE2 search over the last exec's output, `-count=N`),
`cmp stdout file`, `stdin file`, `env K=V`, `exists file`,
`[unix]`/`[windows]`/`[short]` condition guards, `exec cmd &` +
`wait` for background processes.

### 7.2 MySQL server mode tests

The `mysql` custom command runs a query against a server started by the
script and dumps rows to the script's stdout, so `stdout`/`cmp` work on
it (custom commands write via `ts.Stdout()` — supported since
v1.15.0):

```go
func cmdMySQL(ts *testscript.TestScript, neg bool, args []string) {
	// usage: mysql <dsn-env-var> 'query'
	db, err := sql.Open("mysql", ts.Getenv(args[0]))
	ts.Check(err)
	defer db.Close()
	rows, err := db.Query(args[1])
	if neg {
		if err == nil { rows.Close(); ts.Fatalf("query unexpectedly succeeded") }
		return
	}
	ts.Check(err)
	defer rows.Close()
	dumpTabSeparated(ts.Stdout(), rows)
	ts.Check(rows.Err())
}
```

### 7.3 Plugin e2e harness

Goal: per-plugin tests written as data, one shared runner, runnable on
GitHub Actions **without secrets on PRs** (recorded HTTP) and **live on
a schedule** (drift detection). Agreed mode: hybrid.

**Spec file** `plugins/<name>/tests.yaml`:

```yaml
profile:                # becomes the plugin profile config
  token: ${GITHUB_TOKEN}   # env-substituted; in replay mode a dummy value is injected
tests:
  - name: list repositories
    query: SELECT * FROM github_my_repositories LIMIT 5;
    assert:
      min_rows: 1
      columns: [id, name, full_name, private]     # subset match, order-insensitive
      column_types: {id: INTEGER, name: TEXT}
  - name: insert is rejected without required params
    query: SELECT * FROM github_stargazers_from_repository;
    expect_error: "missing.*repository"            # regex on the error
```

Default assertions are invariant-style (`min_rows`, column presence,
types, optional per-column regex) because live API data drifts; `exact`
row assertions exist but only for deterministic sources.

**Runner** (`test/pluginrunner/`): a Go test that, per spec file,

1. builds the plugin (`go build` in its directory — each plugin is its
   own module, e.g. `plugins/github/go.mod`),
2. prepares an isolated `XDG_CONFIG_HOME`, then loads the plugin
   through the existing dev-mode path: run `anyquery -q ... --dev` with
   an init that calls `load_dev_plugin('<name>', '<devManifest.json>')`
   (dev functions registered at `namespace/namespace.go:366-379`,
   manifest format produced by `controller/dev.go`). This avoids
   faking the registry/install pipeline entirely,
3. executes each query via `anyquery -q --format json` and applies the
   assertions to the parsed JSON.

**HTTP record/replay.** Plugins are child processes making their own
HTTPS calls, so the interception point is the environment, not Go
imports: the runner starts a MITM forward proxy and exports
`HTTPS_PROXY=http://127.0.0.1:<port>` + `SSL_CERT_FILE=<generated CA>`
before launching anyquery — child plugin processes inherit the
environment (`exec.Command` without `Env` override,
`rpc/rpc.go:226`), and Go's default `http.Transport` honors both
variables (`ProxyFromEnvironment`, `crypto/x509` root loading). Use
`github.com/elazarl/goproxy` with a test-only generated CA committed
nowhere (generated per run; only the cassettes are committed).

- **Record mode** (`PLUGIN_TEST_MODE=record`): forward upstream, write
  one JSON cassette per request to
  `plugins/<name>/testdata/cassettes/<hash>.json` with key
  `(method, host, path, sorted-query, sha256(body))` and value
  `(status, whitelisted response headers, body)`. **Scrub before
  writing**: drop `Authorization`/`Cookie`/`Set-Cookie` from both
  directions and let the spec declare extra scrub regexes for tokens
  embedded in URLs or bodies.
- **Replay mode** (default): serve from cassettes; a miss fails the
  test with the request line and the three nearest keys (debuggability
  matters more than cleverness).
- Known limitation to document: a plugin that builds a custom
  `http.Transport` without `ProxyFromEnvironment` bypasses the proxy —
  its live tests still work but recording silently captures nothing;
  the replay run then fails loudly on the miss, which is the desired
  signal. Audit the first-party plugins once (grep for
  `&http.Transport{`).

**GitHub Actions.**

- `plugin-tests.yml` (on PR): replay mode, no secrets, matrix over
  plugins whose directory changed
  (`dorny/paths-filter` or `git diff --name-only` in a setup job).
  Deterministic, fork-safe.
- `plugin-tests-live.yml` (nightly + manual dispatch): live mode with
  repo secrets, full matrix, `continue-on-error` per plugin, opens/updates
  a single tracking issue with the failure table, uploads re-recorded
  cassettes as an artifact so refreshing them is one download away.

### 7.4 Interactive/PTY tests (phase 2)

Two escalation levels, both Linux/macOS-only (guard with
`[!windows]` / `runtime.GOOS`):

1. testscript's built-in `ttyin file` + `ttyout 'regex'` (present in
   go-internal since v1.12) runs the next `exec` under a real PTY —
   enough for “the shell renders a prompt and executes a typed query”.
2. For completion/highlighting behavior: `creack/pty` (maintained,
   v1.1.24) + `Netflix/go-expect` + `hinshun/vt10x` (both frozen since
   2022 but small and stable — pin them). The non-negotiable pattern,
   learned the hard way by every project using this stack: feed all
   output into the vt10x screen emulator and **assert on the rendered
   screen with a poll loop**, never on the raw byte stream (go-prompt
   repaints with cursor-movement sequences that make stream matching
   flaky); keep a background `c.ExpectEOF()` goroutine draining (output
   is only pumped during Expect calls); always set
   `expect.WithDefaultTimeout` (default is infinite) and a real window
   size (`pty.Setsize`; 0×0 breaks go-prompt's dropdown rendering).

Do not build level 2 into CI gating until it has run flake-free for a
while locally; PTY tests earn trust slowly.

### 7.5 What NOT to build

- No bespoke test DSL beyond `tests.yaml`; testscript's language covers
  the CLI side.
- No BATS/shell harness (Windows exists).
- No in-repo mock servers per plugin — the proxy/cassette layer is the
  single mocking mechanism.

## 8. Sequencing

Priorities agreed: testing first, then the cache (headline feature),
then parser + shell UX. Cancellation ships in the same protocol bump as
the cache. Excel is independent and can proceed in parallel.

1. **Testing foundation** (§7.1, §7.2): testscript harness + a starter
   corpus of scripts covering current behavior (read_csv, formats,
   dot-commands, server-mode SELECT). These lock in behavior before the
   risky changes. Small PR, no product code touched.
2. **Parser correctness fix** (§5.3, the `modified` flags): tiny diff,
   big win, fully covered by the §5.5 semantic round-trip test. Ship
   independently and early.
3. **Protocol v2 PR** — one atomic breaking change carrying: cache
   (`CachePolicy`, `InitializeArgs.CacheBrokerID`, cache RPC service,
   `cachingClient`), cancellation (`Interrupt` RPC,
   `ReaderInterfaceContext`, plugin map mutexes), and
   `ProtocolVersion = 2`. Plugin authors rebuild **once**. Handshake
   failure against a v1 plugin must produce:
   `plugin X was built for anyquery < 0.5.0; run 'anyquery update X'`
   (wrap the go-plugin error in `rpc.ConnectionPool.NewClient`,
   `rpc/rpc.go:245-247`).
4. **Host-side cancellation** (§4.2, §4.3, fork `IsInterrupted`,
   per-conn vtab fix): depends on 3 for the plugin half but the
   SQLite-level interrupt + KILL + Ctrl-C parts are host-only and can
   land right after.
5. **rqlite/sql adoption** (§5.4) behind the corpus tests.
6. **Shell lexer/completer** (§6): independent of everything else.
7. **`read_xlsx`** (§3): independent; needs the testscript harness for
   its e2e layer.
8. **Plugin e2e harness + CI** (§7.3): after 3, so the reference
   plugins it exercises are already on protocol v2.
9. Stretch: MySQL result streaming (§4.5), `SHOW PROCESSLIST`.

Release notes must call out the three visible breaks: all plugins must
be updated (protocol v2), legacy plugin caches are wiped, Ctrl-C at the
shell prompt no longer exits (Ctrl-D does).
